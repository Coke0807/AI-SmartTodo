# 开发技术简介

## 1.1 项目概述

智能任务管家 (SmartTodo AI) 是一款专为企业综合实训设计开发的支持混合推理模式的智能待办事项看板与任务管理系统。系统核心亮点在于双模驱动的混合推理路由：系统可根据网络状态和用户配置，在云端大模型 API（如 DeepSeek、OpenAI）与本地轻量级模型（如通过 Ollama 部署的 Qwen/Llama 模型）之间进行智能路由切换与无缝降级。系统还集成了基于本地文档内存相似度检索的轻量级 RAG 知识库问答以及全局 AI Copilot 流式助手。系统基于 Go 1.26 后端与 Python 3.12 AI 网关，结合 React 18 前端，通过 gRPC 协议传输 Protobuf 数据，实现高并发、低延迟的混合推理业务闭环，并使用 Docker 进行容器化编排部署。

## 1.2 核心框架与技术栈详解

1)  Go 1.26 与 Python 3.12 (项目核心后端与 AI 语言)

Go Gin 负责高性能的业务开发，Python FastAPI & gRPC 负责双栖 AI 路由与轻量 RAG 接口。二者相辅相成，构建高性能微服务。

2)  React 18 & TypeScript (前端核心框架)

前端使用 React 18 配合 Ant Design 组件库，采用玻璃拟物化暗黑系风格设计。通过 TypeScript 提供类型安全，基于 Vite 构建极速热更新开发环境，完整展示任务看板、AI 配置和 RAG 问答界面。

3)  Go Gin 框架与 GORM ORM (后端与持久化工具)

业务后端基于 Go Gin 搭建，持久层使用 GORM 对 PostgreSQL 16 数据库进行操作，通过 AutoMigrate 自动迁移生成数据表，并使用 Preload 实现主子任务的一对多高效级联查询。

4)  Python FastAPI & gRPC (双栖 AI 网关服务)

AI 智能网关基于 Python FastAPI 提供微服务健康检查，后台运行高并发 gRPC 服务以 Protobuf 协议与 Go 进行低延迟跨语言通信，处理核心的 AI 推理路由和文本检索。

5)  gRPC & Protocol Buffers (高性能跨语言通信)

Go 与 Python 服务之间使用 gRPC 协议进行通信，省去传统 HTTP/JSON 序列化解析开销。接口格式由 proto 文件统一定义，保证了强类型约束与接口的高稳定性。

6)  JWT Token 统一身份鉴权与拦截器

后端编写了 Gin JWTAuth 中间件拦截器。除注册、登录外，所有 API 必须在 Request Header 中携带 JWT Bearer 令牌，经解析校验后提取 userID 注入 Context 以防越权访问。

7)  Ollama & DeepSeek 混合推理路由 (AI 核心引擎)

基于 OpenAI Python SDK 整合本地 Ollama (Qwen) 与云端 DeepSeek API。Hybrid 模式下优先请求云端获取高质量输出，超时或报错则自动无缝秒级切换至本地模型推理，高可用且零成本。

8)  其他技术

针对实训文档，手写了内存级 Chunking 文本切分与基于词频加权排序的相关度评分检索算法，免去部署重型向量数据库的复杂架构，在本地安全隐私下实现秒级文档问答。

全局异常处理中间件：Go 端注册全局 Recovery 中间件，捕获系统 Panic 异常，统一返回友好包装响应，隐藏底层数据库及 gRPC 敏感堆栈信息。

## 1.3 开发环境完整搭建说明

1)  基础环境

Node.js 20、pnpm 9、Go 1.26、Python 3.12 (uv)、PostgreSQL 16 数据库、Ollama、Docker Desktop 运行环境。

2)  环境搭建分步流程

- 本地拉起 Ollama 并 pull 下下载本地轻量大模型 \`qwen2.5:7b\` 或 \`llama3\`。

- 在 Python AI 网关中开发 gRPC 服务器，编写 \`llm_client.py\` 路由模块、\`rag_service.py\` 检索模块、\`split_service.py\` 拆解模块。

## 2.2 功能模块设计

用户与配置模块：实现 JWT 鉴权保护的注册登录与密码哈希；提供每个用户单独配置大模型路由、Endpoint 以及 API 密钥的保存与获取。\
看板任务模块：实现任务的 CRUD，设置任务标题、描述、工时、优先级和截止时间；支持子任务级联操作与看板统计展示。\
AI 混合路由引擎：Python 网关接收 gRPC 调用，支持本地模式、云端模式和 Hybrid 混合模式的推理决策，并进行秒级自动故障降级。\
智能任务拆解模块：利用大模型 JSON Mode 输出结构化任务要素，使用 Pydantic 进行类型断言和属性强校验，集成 fallback 兜底生成逻辑。\

微服务公共保障模块：Protobuf 服务契约生成、Go 端 gRPC 客户端连接池、Gin 全局异常 Recovery 中间件。

## 2.3 系统整体分层架构

项目采用标准Spring MVC五层分层架构，代码按包划分，职责分离，耦合度低：

- React 18 前端表现层：基于 React 框架和 Ant Design 库，处理用户页面交互，通过 Axios 向后端发送请求，展示看板拖拽、AI 路由配置面板、流式 Copilot 聊天框等。

- Go Gin 业务控制层：负责接收 HTTP 请求，处理用户身份 JWT 校验，执行任务的数据库事务操作；同时作为 gRPC 客户端，根据用户配置将 AI 请求高效序列化并路由至 AI 网关。

- Python gRPC/FastAPI AI 网关业务层：提供健康检查接口；通过 gRPC 后台服务接收 Go 端的调用，执行核心的混合路由决策、结构化拆解校验、RAG 内存检索及大模型 API 调用。

- PostgreSQL 16 数据库持久层：使用关系型数据库 PostgreSQL，结合 Go 的 GORM 进行表结构自动创建，提供 users, todos, sub_tasks 三张核心数据表的高效持久化。

- Protobuf 协议传输层：通过定义 \`todo_ai.proto\` 接口，定义了微服务间的通信契约（包含 AIConfig、SplitTask、Chat、RAGQuery 接口），提供二进制、强类型的服务调用桩。

- 微服务保障包：包含 Go 的全局异常捕获、BCrypt 密码加密器、Python 的 RAG 内存相似度打分器、物理隔离多租户目录创建器等支撑组件。

## 3.2 表与表之间的关联关系

1)  一对多关系：\
    1. 用户表 (users) -\> 待办事项表 (todos)：一个用户可以拥有并管理多个独立的待办事项。\

## 3.3 数据库整体设计规范

- 外键与索引：外键列均显式声明 Index 索引，提升 GORM Preload 关联查询效率。\
  时间精度：使用 timestamp 时区敏感格式自动记录创建与更新时间。\
  灵活配置：用户 AI 大模型路由配置采用非结构化 text（JSON 字符串）列，极大地提高了路由字段扩展灵活性，避免频繁更改表结构。\

###### 4.1.1 注册与登录

系统采用基于 JWT 的身份鉴权体系代替传统的 Session 会话存储。在用户注册阶段，前端收集用户名和密码，通过 API 异步发送至 Go 后端。后端利用 GORM 框架查询 users 表，通过用户名索引快速校验是否已存在同名账号。为保证密码存储安全，后端利用 Bcrypt 算法对用户明文密码进行单向哈希加密，将其转化为不可逆的密文哈希串并存入数据库，彻底防范了数据库泄露带来的安全风险。在用户登录阶段，系统将前端传入的明文密码与数据库中查询出的哈希密文进行比对，验证成功后使用 Go 后端生成包含用户 ID 信息的 JWT 令牌（Token）并返回给前端，前端将该 Token 保存在本地 localStorage 中作为后续所有请求的凭证，从而在微服务架构下保证了数据流的安全可靠。

## 4.3 AI混合推理配置管理实现

AI混合推理配置管理模块提供本地 Ollama 路由、云端 API 路由配置、以及 Hybrid 智能模式下的秒级降级切换配置界面与业务逻辑。

###### 4.3.1 混合推理 API 路由实现（Python）

AI 智能路由网关使用 Python 语言承接，作为独立的异构微服务。AI 网关的 llm_client 模块整合了本地 Ollama 大模型（推荐部署 Qwen2.5:7b 局部模型）与云端 OpenAI/DeepSeek 兼容的接口。在 Hybrid 智能混合路由模式下，系统会优先向云端的高智能模型发起请求以获取最专业、全面的语义生成。若遭遇网络瞬间断开、超时或大模型接口限流报错，智能路由模块会利用 Python 的 Try-Except 异常捕获机制，自动无缝且零延迟地降级切换至本地部署的 Ollama 模型进行本地推理，保障了用户看板系统的 AI 服务 100% 始终可用。

运行效果：用户在"AI混合推理配置"页面中实时切换运行模式。在 Hybrid 模式下，大模型调用请求优先路由至云端 API。若人为拔掉网络线或发生超时，系统自动秒级降级至本地 Ollama Qwen，业务体验零中断。

###### 4.3.2 混合流式传输（SSE）缓冲降级机制（Python）

大语言模型对话支持流式传输（SSE）打字机效果。然而在智能混合模式流式传输中，如果云端连接在中途断开，直接切换降级会导致前半段是云端回答，后半段是本地回答，或者导致 JSON 格式在交界处破碎。为此系统首创设计了"Token 缓冲区机制"：AI 网关在 Hybrid 流式推送时，先在内存中对云端大模型吐出的 Token 块进行全量缓冲；只有当云端流式完全成功读取且无报错后，再一次性向前端客户端 flush 缓冲的所有 Token；若云端在生成中途断开，系统将直接丢弃内存缓冲，并自动从头重新调起本地 Ollama 服务重新流式推送，给用户带去极佳的交互防破碎体验。

运行效果：点击流式对话时，AI 助手以丝滑打字机效果吐出字符。即使云端网络在生成中途断开，客户端也不会看到破碎内容，系统会自动重新拉起本地 Ollama 重新生成并无缝衔接。

## 4.5 RAG本地知识库问答与全局 AI Copilot（创新特色）

本项目集成阿里云百炼平台（Ollama & DeepSeek大模型），实现商城专属智能客服，是系统的亮点功能。

###### 4.5.1 RAG 本地文本分片与内存相关度检索算法（Python）

RAG 本地知识问答是系统的重要特色。为简化实训系统的部署复杂度，免去部署重型商业向量数据库，系统手写了一套内存级零依赖文本检索打分算法。当用户上传 txt/md 文档时，Python 网关对文本利用正则表达式以空行为界进行 Chunking（文档切分）。接着提取用户提问的关键词集合，对每个 Chunk 进行词频共现重合度计算，并加入文档长度惩罚因子进行相关度评分。最终排序提取打分最高的 Top 3 文本片段组装成 Context 组，强制使用本地 Ollama 驱动，既实现了极高精度的知识检索，又 100% 保证了用户笔记的隐私安全与本地免费处理。

###### 4.5.2 RAG 智能问答与 Chat 流式 gRPC 接口实现

AI 网关的 gRPC 服务端接收到来自 Go 端转发的问答请求后，将其路由给本地 RAG 服务。RAG 服务将检索得到的知识背景片段注入提示词中，并调用 execute_llm 执行本地大模型推理。Go 端控制器 ai.go 接收到 gRPC 响应后，将结果包装为 JSON 响应反馈给前端 React 看板组件，React 通过无刷新追加，将人机对话文本渲染在全局 AI Copilot 悬浮对话气泡框中，使用户可以在管理任务看板时，随时调起悬浮框向 AI 获取笔记解答与通识答疑。

## 4.6 跨语言 gRPC 通信与多租户文件隔离上传实现

###### 4.6.1 全局异常处理器与响应统一包装

系统在 Go Gin 端注册了全局拦截 Recovery 中间件。在处理 HTTP 路由的 Goroutine 发生运行时 Panic（如数据库连接断开、空指针异常等）时，中间件捕获 Panic 异常，并在后台日志记录堆栈追踪，同时对前端统一返回格式化 JSON 的 500 服务器错误友好提示。这一设计彻底避免了底层 PostgreSQL 数据表名、gRPC 连接细节或物理文件路径直接在客户端明文暴露，在软件工程层面上大大提升了系统的安全性、健壮性与稳定性，起到了绝佳的安全锁防范机制。

运行效果：当用户上传超过 2MB 的文本或上传不支持的后缀格式时，后台拦截并统一抛出友好错误提示；文档上传落盘时按 userID 动态分区隔离，杜绝租户间越权读取。

###### 4.6.2 多租户物理隔离文件上传实现

用户在使用 RAG 知识库上传学习笔记时，Go 后端通过 FormFile 接收文件，校验文档后缀白名单（仅限 md、txt）及 2MB 的大小限制。之后，后端利用当前会话中解析出的 userID，动态隔离创建物理子目录 \`uploads/user\_{userID}\`。所有文档在存储、读取、检索时均完全通过该用户目录进行限制。这在多租户物理存储层面筑牢了数据安全屏障，从物理存储层彻底隔离了租户数据越权的隐患，实现底层物理级安全隔离与防越权读取。

运行效果：用户上传 RAG 笔记后，可针对该笔记内容进行自然语言问答。AI 网关会智能分析检索切片，结合上下文给出准确解答。若问及笔记无关的问题，AI 将明确拒绝。
